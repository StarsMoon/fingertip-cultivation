# 《指尖修仙》项目开发流程规划

> 项目经理视角的端到端开发流程，从脚手架到上线到持续运营

---

## 0. 项目管理哲学

| 原则 | 说明 |
|---|---|
| **Playtest-Driven** | 每个里程碑结束必须可玩，不做「代码写完但玩不了」的阶段 |
| **垂直切片先行** | 先做出 1 个功法 + 1 种妖魔的完整闭环，再横向扩展 |
| **数值即玩法** | 功法平衡不是后期调优，是核心设计，从第一天就开始测 |
| **性能即体验** | 60fps 是硬指标，任何导致掉帧的提交直接回滚 |
| **小步快跑** | 每周内部试玩，每 2 周外部测试，快速验证快速改 |

---

## 2. Phase 0 — 基建 & 原型 (Week 1)

### 目标

> 跑通「微信开发者工具里出现一个 Canvas，触摸移动一个方块」的全链路

### 任务分解

| # | 任务 | 交付物 | 工时 | 优先级 |
|---|---|---|---|---|
| 0.1 | 项目脚手架搭建 | TS 编译 + webpack + 微信小游戏适配 + 热更新 | 1d | P0 |
| 0.2 | 游戏主循环框架 | Game.ts 主循环 (requestAnimationFrame) + 固定时间步长 | 0.5d | P0 |
| 0.3 | 触摸输入系统 | 单指拖拽 → 移动一个方块 | 0.5d | P0 |
| 0.4 | Canvas 渲染器 v1 | 清屏 + 绘制矩形 + 坐标系 | 0.5d | P0 |
| 0.5 | 实体基类 | Entity.ts: position, velocity, update, render | 0.5d | P0 |
| 0.6 | Git 工作流确立 | 分支策略 + commit 规范 + CI 脚本 | 0.5d | P1 |
| 0.7 | 调试工具 | FPS 计数器 + 实体数量显示 + 性能面板 | 0.5d | P1 |

### 质量门禁

- [ ] 微信开发者工具中运行，触摸移动方块
- [ ] TypeScript 零编译错误，strict 模式
- [ ] 稳定 60fps（调试面板验证）
- [ ] webpack 构建 + 热更新正常
- [ ] `git push` 触发 CI（至少做 lint + build 检查）

### 技术决策（Phase 0 必须确定）

| 决策项 | 需确定内容 | 影响 |
|---|---|---|
| 坐标系 | 逻辑坐标 vs 物理像素，适配策略 | 全局，后续改成本极高 |
| 时间步长 | 固定 dt 还是可变 dt | 物理一致性 |
| 渲染策略 | 每帧全清重绘 vs 脏矩形 | 性能基线 |
| Canvas 尺寸 | 设计分辨率 + 适配方案（letterbox/fill/fit） | 所有 UI 布局 |

### 功法验收标准

每增加一个功法，必须通过以下 checklist：

- [ ] 获取后立刻可感知（视觉/行为变化）
- [ ] 进化后变化明显（不是只加数值）
- [ ] 与至少 1 个其他功法有化学反应（1+1>2）
- [ ] 不与其他功法冲突（叠加后不崩不卡）
- [ ] 音效区分度高

## 8. Git 工作流

### 分支策略

```
main (生产分支，始终可发布)
  │
  ├── develop (开发集成分支)
  │     │
  │     ├── feature/combat-core      ← 功能分支
  │     ├── feature/skill-system
  │     ├── feature/boss-fox
  │     └── fix/projectile-penetration
  │
  ├── release/v0.1                   ← 发版分支
  └── hotfix/crash-on-low-memory     ← 紧急修复
```

### Commit 规范

```
<type>(<scope>): <subject>

type: feat | fix | perf | refactor | docs | test | chore | data
scope: combat | skill | ui | audio | render | data | platform

示例:
feat(combat): implement enemy spawner with time-curve generation
fix(skill): chain lightning not jumping to offscreen enemies
perf(render): add spatial hash for collision detection
data(skill): rebalance fireball damage from 15 to 12
```

### 里程碑 Tag

```
v0.1.0-alpha  ← Phase 1 结束：核心战斗可打
v0.2.0-alpha  ← Phase 2 结束：功法+养成闭环
v0.3.0-beta   ← Phase 3 结束：外部测试版
v1.0.0        ← Phase 4 结束：正式上线
```

## 9. 测试策略

### 测试金字塔

```
          ┌──────────┐
          │ E2E 测试  │  ← 关卡流程自动化（最少，最贵）
          │  5%      │
         ┌┴──────────┴┐
         │  集成测试    │  ← 功法组合/碰撞/生成（关键路径）
         │   15%      │
        ┌┴────────────┴┐
        │   单元测试     │  ← 数值公式/对象池/数学工具（最多，最便宜）
        │    80%       │
        └──────────────┘
```

### 各阶段测试重点

| Phase | 测试重点 | 方法 |
|---|---|---|
| Phase 0 | 构建/编译/运行 | CI 自动化 |
| Phase 1 | 碰撞正确性、敌人 AI 行为、弹幕生命周期 | 单元测试 + 手动试玩 |
| Phase 2 | 功法叠加无冲突、进化数值正确、Boss 行为 | 集成测试 + 试玩 |
| Phase 3 | 全机型性能、排行榜正确性、分享图生成 | 兼容性测试 + 外部测试 |
| Phase 4 | 广告奖励到账、合规项、极端场景 | 回归测试 + 审核模拟 |

### 性能测试标准

| 场景 | 目标 | 测量方式 |
|---|---|---|
| 50 敌人 + 50 弹幕 | 60fps 稳定 | FPS 计数器 |
| 200 敌人 + 200 弹幕 | ≥ 55fps | FPS 计数器 |
| 境界突破特效 | 无卡顿 | 单帧耗时 < 33ms |
| 首次加载 | < 2s | 冷启动计时 |
| 连续 10 局 | 无内存泄漏 | 内存监控 |

## 10. 风险管理

### 风险看板

| 风险 | 概率 | 影响 | 预防措施 | 应急方案 |
|---|---|---|---|---|
| 全屏弹幕掉帧 | 🔴高 | 🔴毁灭 | 对象池+粒子上限+分级渲染 | 降低同屏上限，减少特效 |
| 功法数值失衡 | 🔴高 | 🟡严重 | 数据驱动+每周数值评审 | 热更新紧急调数值 |
| 微信审核被拒 | 🟡中 | 🟡严重 | 提前研读审核规范+自查清单 | 按拒审原因快速修改重提 |
| 包体超 4MB | 🟡中 | 🟢中等 | 资源按需下载+压缩 | 美术降级/音频降采样 |
| 美术资源延期 | 🟡中 | 🟢中等 | 先用占位符开发，后期替换 | 用纯几何图形+粒子替代精灵 |
| 留存率不达标 | 🟡中 | 🔴毁灭 | 每周试玩+数据驱动迭代 | 紧急加每日挑战/活动 |
| 技术方案踩坑 | 🟢低 | 🔴严重 | Phase 0 充分验证+技术预研 | 降级方案（如降帧到30fps） |

### 关键决策点

| 时间点 | 决策 | 依据 | 回退方案 |
|---|---|---|---|
| Week 1 末 | 确认技术方案可行 | Phase 0 质量门禁 | 换渲染方案（如改用 PixiJS） |
| Week 3 末 | 确认核心玩法成立 | 内部试玩 #1 反馈 | 大改战斗机制 |
| Week 5 末 | 确认功法深度足够 | 内部试玩 #2 反馈 | 砍功法数量，保质量 |
| Week 7 末 | 确认可进入上线准备 | 外部测试 NPS ≥ 7 | 延期 1 周打磨 |

## 11. 每周节奏

### 日常

```
09:00  站会 (15min): 昨天做了什么 → 今天做什么 → 有什么阻塞
       无阻塞则快速过，有阻塞当场决定方案

全天   开发/测试

18:00  提交代码 → CI 跑通 → 当日进度更新到项目看板
```

### 周节奏

| 日期 | 活动 | 目的 |
|---|---|---|
| 周一 | Sprint Planning | 确认本周目标，拆分任务 |
| 周三 | Mid-week Playtest | 试玩当前版本，快速发现问题 |
| 周五 | Sprint Review + Retro | 演示成果 + 回顾改进 |

### 周交付物

每周五必须产出：

1. **可玩版本** — 打包到微信开发者工具可直接运行
2. **周报** — 本周完成 / 下周计划 / 风险项 / 需要决策的问题
3. **数据**（Phase 3 起）— 试玩数据 + 埋点数据初报
